# 서버시작.bat 에러 이슈 노트

---

## 발생 날짜
2026-04-14

---

## 증상
- `서버시작.bat` 실행 시 **UAC 팝업** 뜨면서 `Install.exe` 설치 요청
- 팝업 내용: `C:\Program Files (x86)\IncrediBuild\Install.exe at: https://nodejs.org`
- bat 실행해도 창이 바로 사라지고 서버가 뜨지 않음

---

## 원인 분석 (삽질 과정)

### 1차 오해 — Node.js 문제?
- `where node` 명령어 결과가 비어있어서 Node.js PATH 문제로 오해
- 사실: PowerShell을 재시작하지 않아서 새 PATH가 반영 안 된 것
- `Get-Command node` 로 확인하니 `C:\Program Files\nodejs\node.exe` v24.13.0 정상 존재

### 2차 오해 — IncrediBuild 레지스트리 문제?
- IncrediBuild가 자동실행 등록된 줄 알고 레지스트리 탐색
- 레지스트리에는 IncrediBuild 자동실행 항목 없음
- 스케줄러에도 없음

### 3차 오해 — IncreBuild 업데이트 타이밍 문제?
- IncrediBuild가 Node.js 설치 시 함께 딸려온 번들 프로그램
- 업데이트를 bat 실행 타이밍에 시도한 것으로 추정
- IncrediBuild 업데이트 후에도 동일 증상 지속

### 진짜 원인 — bat 파일 인코딩 깨짐
- bat 파일을 Claude가 수정하는 과정에서 **파일 인코딩이 깨짐**
- bat 파일이 UTF-8로 저장되면서 Windows cmd가 읽지 못하고 명령어들이 글자 단위로 쪼개져 실행됨
- 오류 메시지: `'erver'은(는) 내부 또는 외부 명령이 아닙니다` ('S'erver — 첫 글자 짤림)

---

## 최종 해결

bat 파일을 **처음부터 새로 작성** (인코딩 깨진 파일 덮어쓰기)

핵심 변경사항:
- `node save-server.js` → `"C:\Program Files\nodejs\node.exe" save-server.js` (전체 경로 직접 지정)
- 불필요한 Node.js 탐색 로직 전부 제거
- 파일을 깔끔하게 새로 Write해서 인코딩 문제 해결

최종 bat 내용:
```bat

chcp 65001 >nul
cd /d "%~dp0"
echo.
echo  ==========================================
echo   HTML Presentation Server
echo  ==========================================
echo.
echo  서버 시작 중...
echo.
start http://localhost:3000/presentations/prompt_engineeringv02/prompt_lecture.html
"C:\Program Files\nodejs\node.exe" save-server.js
pause
```

## 교훈

1. **bat 파일 수정 시 인코딩 주의** — Claude가 Edit 도구로 수정하면 인코딩이 깨질 수 있음. 반드시 Write로 새로 작성할 것.
2. **where node 는 PowerShell 재시작 전엔 안 됨** — 환경변수 추가 후 반드시 터미널 재시작 필요.
3. **node.exe 전체 경로 지정이 가장 안전** — PATH 의존 없이 확실하게 실행 가능.
4. **IncrediBuild는 Node.js 설치 시 딸려오는 번들** — Visual Studio 사용자라면 그냥 둬도 됨. 팝업은 IncrediBuild 업데이트 타이밍 문제.

# Export 자립성 대수술 (v2.0)

## 발생 날짜
2026-04-24

## 증상
- Save As 한 파일을 다른 PC에 가져가면 전혀 작동 안 함.
- Export 한 파일도 다른 PC에서 깨짐 — CSS, 영상, 이미지 모두.
- PPTX 폴더 전체를 복사해가야만 작동 (그마저도 상대경로 의존).

## 원인 (복합)
1. **미디어 미인라인** — exportHTML()은 CSS/JS만 인라인, 이미지/비디오는 상대경로 방치.
2. **폰트는 Google CDN 의존** — 오프라인 PC에서 한글 폰트 깨짐.
3. **`_inlineAssets` 폴백이 절대경로 박음** — fetch 실패 시 `file:///C:/...` 형태로 href 재작성 → 다른 PC에서 당연히 실패.
4. **`save-server.js` 파일이 아예 없었음** — 에러이슈노트에서만 언급되고 실제 파일은 유실 상태였음.
5. **blob: URL 미디어** — 에디터에서 드롭한 이미지가 Export에 포함 안 됨.

## 해결 (v2.0)
- **save-server.js** 신규 작성 (3000/3001 포트, /save /save-as /export /ping).
- **exportHTML()** 완전 재작성 — 폴더 기반 flat 복사:
  - 서버 생존 확인 → 폴더 경로 프롬프트 → 미디어 전수 스캔 → 상대경로를 basename으로 재작성 → 서버에 복사 요청 → manifest 생성.
  - 미디어는 임베드 안 함 (원본 복사). 폰트만 base64 임베드.
  - editor.js 완전 제거 (뷰어 전용).
- **`_validateExportedHTML`** 자체검증 — localhost/file://, 절대경로, `../` 잔존 감지.
- **`_export_manifest.json`** 자동 생성 — 복사된 파일 목록 + 누락 기록.
- **`engine/EXPORT_CONTRACT.md`** 신규 — 향후 재발 방지 계약 문서.
- Save As는 로컬 전용 (서버 필수).

## 향후 주의사항
- **새 미디어 속성 추가 시** `exportHTML()` 내의 미디어 수집 루프(7a/7b/7c)를 반드시 갱신.
- **CSS에 `url(...)` 절대 추가 금지** — 폰트 외에는.
- **매 릴리즈마다 EXPORT_CONTRACT.md의 테스트 절차 실행** (다른 폴더/오프라인 검증).
- Export 실패 시 **절대** fallback 경로(download, absolute path embed)를 만들지 말 것 — 과거에 이게 망친 원인.

# UX 전환: prompt → showDirectoryPicker (v2.1)

## 발생 날짜
2026-04-24

## 증상
- v2.0 Export가 기능은 되는데 **사용자 경험이 처참**했음.
- `prompt()`로 한글 경로를 직접 타이핑하게 만듦 → URL 인코딩된 쓰레기 기본값, 키보드로 `C:\temp\...` 타이핑.
- 형 피드백: "기본적인 것을 왜 새롭게 만드느냐. 화장실 가서 똥 싸고 안 닦고 나온 격."

## 원인 (근본적 설계 실수)
1. **제약을 핑계 삼음** — `file://` 환경에선 File System Access API가 막힌다는 걸 알고 "prompt()로 가자"로 결정. 하지만 **localhost 서버가 이미 돌고 있으면 `http://localhost`에서 열리므로 API 사용 가능**. 전제 자체가 틀렸음.
2. **UX 여정을 글로 서술해보지 않음** — 코드 로직만 짜고 "사용자가 버튼부터 완료까지 어떤 창을 몇 번 보는가"를 머릿속으로라도 시뮬레이션 안 함.
3. **"기능 작동 = 완료"로 착각** — 테스트가 통과해도 사람이 쓰기 불편하면 미완성이다.

## 해결 (v2.1)
### 아키텍처 전환
- **`file://` 직접 열기 폐기** → 무조건 `http://localhost:3000` 경유.
- `서버시작.bat` 실행 시 크롬 자동 오픈 (`start http://localhost:3000/`).
- 루트 경로(`/`)에서 `index.html` 랜딩 페이지 서빙.
- `index.html` — 프레젠테이션 카드 목록, 서버 상태 배지, 새로고침 버튼.

### Export UX 재작성
- `prompt()` 완전 제거 → `window.showDirectoryPicker({mode:'readwrite'})` 사용.
- 사용자는 **Windows 기본 "폴더 선택" 다이얼로그**로 클릭해서 폴더 선택.
- 파일명은 별도 커스텀 다이얼로그(`inputDlg`) — 기본값 자동 채워짐, Enter로 확인.
- 경로 문자열을 서버에 보내는 대신, 브라우저가 받은 `FileSystemDirectoryHandle`에 **직접** HTML·미디어·manifest를 씀.
- 서버는 `/read-asset`로 원본 미디어를 base64로 제공하는 역할만 함 (HTML 기준 상대 참조 해석 지원).

### Save As도 동일 UX
- `prompt()` 두 번 → `showDirectoryPicker` + `inputDlg` 한 번씩.

## 교훈
1. **Windows/macOS/Linux 모두 "파일 저장" 다이얼로그가 존재한다** — 새로 만들려 하지 말 것. 브라우저에선 File System Access API가 있다.
2. **사용자 UX 여정을 글로 먼저 서술하라** — "버튼 클릭 → 다이얼로그 A → 선택 → 입력 → 완료". 이걸 코드 전에 텍스트로 쓰면 허점이 드러난다.
3. **"완료" 보고 전 직접 써봐야 한다** — 테스트 통과 ≠ 사용자가 만족. 사용자 경험을 머릿속으로 한 번 시뮬레이션.
4. **제약이 있다고 느끼면 한 번 더 의심할 것** — "file://에선 API 막힘" → "localhost면 열려있음". 전제를 의심 안 하면 수준 낮은 해결책으로 빠진다.

## 향후 협업 프로토콜 (형과 합의)
- 기능 착수 전 "사용자 여정"을 글로 서술하고 형 확인받기.
- "제약 때문에 안 된다"고 말하면 형이 "정말? 지금 환경 다시 봐봐" 한번 되물어주기.
- 기능 구현 후 형이 실제로 한 번 눌러보고 피드백 → 그 이후에 "완료" 선언.

# v2.1 후속 수정 — 다이얼로그 과잉 제거 + 한글 파일명 디코딩

## 발생 날짜
2026-04-24

## 증상
Export 누르니 **4단계**를 거쳐야 완료:
1. 폴더 선택 다이얼로그
2. 크롬 "파일 수정 허용하시겠습니까?" 권한
3. "Export HTML 파일명:" 입력 다이얼로그 — **URL 인코딩된 `%EB%AF%B8...` 쓰레기 기본값**
4. "Export 완료" 결과 리포트 모달

형의 기대: "폴더 고르면 그냥 끝나야지. 왜 3개를 더 물어봐?"

## 원인
1. **`decodeURIComponent` 누락** — `location.pathname.split('/').pop()` 은 URL 인코딩 상태. 한글 파일명이 `%EB%AF%B8%EB%93%9C%EC%A0%80%EB%8B%88...` 로 보임.
2. **불필요한 확인 단계를 "친절함"으로 착각** — 파일명 재확인, 결과 리포트 모달 등을 "정보 제공"이라 생각했으나 사용자 입장에선 **방해**.

## 해결
- 파일명 입력 다이얼로그 **삭제**. 현재 HTML 파일명(`decodeURIComponent` 적용)을 자동 사용.
- 결과 리포트 모달 **삭제** (깨끗한 Export 시). 토스트 한 줄로 끝.
- 경고/누락 있을 때만 모달 (사용자가 알아야 할 때만).
- 최종 흐름: `[Export 버튼] → [폴더 선택] → [크롬 권한 허용] → [토스트]` = 실 단계 2번.

## 교훈 (재발 방지)
- **사용자가 정보를 "볼 수 있어야 한다"는 생각과 "매번 봐야 한다"는 생각은 다르다.**
  정보는 manifest 파일에 기록하면 충분. 성공 시 방해하지 말 것.
- **URL 인코딩 디코딩은 경로 다루는 모든 지점에서 체크.** `location.pathname`, `location.href`, `URL(...)` 등등.
- **"확인받기"를 남발하지 말 것.** 매 동작마다 확인받으면 UX 망친다. 실패 가능성 있거나 되돌릴 수 없을 때만.
- **구현 후 형이 실제로 쓴 스크린샷을 받아보기 전까지 "완료"라 하지 말 것** — 이번에도 형이 4단계 스크린샷 찍어서 보내줘서야 문제를 발견했다.

# 미디어(이미지/영상) 드래그 이동 점핑 — 재발 주의

## 발생 날짜
2026-04-24 (오른쪽 패널 리팩토링 v3 검증 중)

## 증상
- 이미지/영상을 슬라이드에 **새로 추가한 직후** 드래그해서 이동하면, 마우스 위치와 상관없이 **엉뚱한 좌표로 순간이동**.
- 특히 가이드 영역 위쪽(패널 바깥)까지 튕겨 나가는 경우 많음.
- 이미 한 번 이동한 미디어는 정상 동작. 최초 이동 시에만 점핑.

## 반복되는 이유 (중요 — 재발 시 제일 먼저 여기부터 확인)
1. **미디어 최초 삽입 시 CSS**: `transform: translate(-50%, -50%)` 로 화면 중앙 정렬. `left/top` 은 비어 있음.
2. **드래그 로직이 요구하는 것**: `oL = parseFloat(style.left) || 0`, `oT = parseFloat(style.top) || 0` → 이때 값은 **0/0**.
3. **점프의 정체**: 실제 화면 위치는 transform 때문에 중앙이지만 `oL/oT` 는 0. mouse delta 를 0 기준으로 더하면 왼쪽 위로 튕긴다.
4. **해결의 핵심**: 드래그 *시작 직전에* `transform='none'` 으로 바꾸고 `getBoundingClientRect()` 로 실제 px 좌표를 계산해 `left/top` 에 넣는다. **그 뒤에** `oL/oT` 를 읽어야 한다.

## 드래그 핸들러가 두 벌 있는 것이 재발 원인
- `engine/editor.js` 에는 드래그 경로가 **두 가지**다:
  - (A) `startDrag(el, e)` — 848 라인. 레거시/내부 이동용. 여기엔 transform→px 변환 로직이 있었다.
  - (B) `startMoveDrag(anchorBlock, e)` — 434 라인. 블럭 시스템 v2 가 사용하는 이동 경로. **여기엔 transform 변환이 빠져 있었음**.
- 패널 리팩토링으로 미디어 클릭 → `startMoveDrag` 경로를 타면서 점핑이 재현됐다.
- **양쪽에 변환 로직이 있어야 한다.** 한쪽만 고치면 또 재발한다.

## 해결 (2026-04-24)
1. `startMoveDrag` 의 `initial = targets.map(...)` 안에서 media-wrap 인 경우 transform → px 변환을 먼저 수행한 뒤 `oL/oT` 를 읽도록 변경.
2. mousedown 핸들러의 "media wrap — 이동 키 없으면 선택만" 분기를 `setBlockState + showBar + startMoveDrag(mw, e)` 로 변경 — **alt 키 없이도 그냥 드래그로 이동** 가능하게.
3. 리사이즈 핸들·X 버튼·영상 컨트롤 UI 클릭은 `startMoveDrag` 를 호출하지 않도록 `closest('.ed-resize-handle, .ed-media-del, ...')` 가드 추가.

## 재발 방지 체크리스트
- 새 드래그 경로를 만들 때마다 **반드시** 다음 순서를 지킨다:
  1. media-wrap 이면 `getBoundingClientRect()` 기반으로 `left/top` 계산해 style 에 주입.
  2. `transform='none'` 으로 중앙 정렬 해제.
  3. **그 뒤에** `oL/oT = parseFloat(style.left/top)` 으로 초기 좌표 읽기.
- 드래그 함수 하나만 고치면 안 된다. `startDrag`, `startMoveDrag`, 그리고 나중에 추가될 모든 새 핸들러 전부.
- **체크 명령**: `grep -n "oL=parseFloat(.*left)" engine/editor.js` 가 여러 건 나오면 각 자리마다 transform 변환이 앞에 있는지 확인.
- 이미지/영상을 슬라이드에 새로 삽입 → 바로 드래그 시험은 회귀 테스트의 필수 시나리오.

## 교훈
- **같은 기능을 담당하는 코드가 두 벌 이상이면 한쪽 수정이 반드시 다른 쪽에서 재발한다.** 경로 통합 또는 최소한 공통 유틸 함수로 추출할 것.
- "이미 고친 버그가 다시 나타난다" = 대부분 **다른 경로에 같은 버그가 또 있다**는 뜻. 같은 증상 = 같은 원인이라 단정하지 말고 호출 경로부터 확인.
- 최초 삽입 직후 상태(아직 style 안 넣은 상태)는 다른 상태와 달라서 버그가 숨기 쉽다 — 테스트 시나리오에 반드시 포함.

# 이슈 #3: Save As 기본 파일명이 URL 인코딩된 상태로 표시됨 (2026-04-24)

## 증상
- `Save As` 실행 시 다이얼로그의 기본 파일명이 `%EB%AF%B8%EB%93%9C%...` 같은 URL 인코딩 문자열로 표시됨.
- 한글 파일명이 포함된 HTML에서만 발생.

## 원인
- `saveAs()` 함수의 2137번 줄 `const defName=(location.pathname.split('/').pop()||'presentation.html');`
- `location.pathname` 은 브라우저 규약상 **URL 인코딩된 상태**로 반환됨.
- 한글 파일명은 `%EB%...` 형태가 그대로 들어감.
- `exportHTML()` 쪽(1639번 줄)은 이미 `decodeURIComponent(...)` 로 처리 중이었음 — **같은 버그가 saveAs 에만 남아있었던 것**.

## 해결 (2026-04-24)
1. `saveAs()` 의 `defName` 계산을 `decodeURIComponent(...)` 로 감싸고 try/catch 로 fallback 처리.
2. 패턴 자체가 재발 위험 있음 → 체크리스트 추가.

## 재발 방지 체크리스트
- `location.pathname`, `location.href`, URL 객체에서 파일명·경로 추출할 때는 **반드시** `decodeURIComponent` 를 먼저 적용한다.
- 검사 명령: `grep -n "location.pathname" engine/editor.js` 결과에서 `decodeURIComponent` 가 **감싸지 않은 줄**이 있는지 확인.
- 파일명 관련 로직을 새로 추가할 때 한글/스페이스/특수문자 파일명으로 회귀 테스트 필수.

## 교훈
- **같은 기능이 여러 함수에 복붙된 경우 한쪽만 고치면 다른 쪽에서 재발.** (Issue #2 와 동일 패턴)
- `location.pathname` 은 인코딩된 값이라는 사실이 직관적이지 않아 반복 실수하기 쉽다 — 유틸 함수로 감싸두는 게 안전.

# 이슈 #4: 영상이 휠 스크롤 전에 자동 재생됨 (2026-04-24)

## 증상
- 슬라이드에 영상을 삽입하고 페이지 진입 시, 휠 스크롤로 블럭들이 나타나기 **전에** 영상이 먼저 재생됨.
- 텍스트/제목 블럭은 `data-step` 순서대로 나오는데, 영상만 시작부터 보임.

## 원인
1. `addMedia()` 가 `ed-media-wrap` 에 `data-step` 을 **안 걸었음** → step 시스템 바깥.
2. CSS `[data-step]` 선언부에만 `opacity:0; pointer-events:none` 이 걸려있음 → `data-step` 없으면 처음부터 보임.
3. `mkVid()` 가 기본 `autoplay=true` 로 설정 → 영상이 DOM에 들어오자마자 재생 시작.

## 해결 (2026-04-24)
1. `mkVid()` 기본 `autoplay=false` 로 변경. 자동재생이 필요하면 오른쪽 패널의 "자동재생 ON" 토글로 명시적 설정.
2. `addMedia()` 에서 삽입 시 현재 슬라이드의 max data-step + 1 을 자동 부여.
3. `autoStepBySlide(slide)` 함수 신규 추가 — 슬라이드의 모든 최상위 블럭(`[data-step]` 또는 `.ed-media-wrap`) 을 **Y 좌표 오름차순**으로 정렬 후 1,2,3... 재배정. 같은 Y 영역(±40px)은 같은 step (동시 등장).
4. 패널 'none' 컨텍스트에 `SEC_STEP_ORDER` 섹션 신설 — "↕ Y좌표 기준 재배정" 버튼으로 수동 호출.

## 재발 방지 체크리스트
- 새로운 블럭(미디어 포함)을 DOM 에 삽입하는 함수는 **반드시** `data-step` 을 부여한다. 안 그러면 step 시스템 바깥이 되어 항상 보임.
- 미디어 기본값(`mkVid`/`mkImg`/`mkIframe`) 은 `autoplay=false` 유지. 사용자가 명시적으로 켜야만 자동재생.
- 블럭 위치를 바꾼 뒤엔 반드시 "재배정" 버튼 권장.
- 체크 명령: `grep -n "data-step" engine/editor.js | grep -E "setAttribute|dataset.step"` 로 부여 지점 전수 확인.

## 교훈
- **step 시스템에 포함되지 않은 요소는 "기본 상태"가 "보임"** — 한 번 더 강조해야 할 설계 원칙.
- 미디어 삽입 로직은 텍스트 블럭 삽입 로직과 다른 경로(wrapMedia)를 거쳐서 data-step 처리가 누락되기 쉬움. 경로 통합 또는 공통 훅 필요.
- `autoplay` 는 UX 차원에서 기본 OFF 가 안전. 재생 순서를 step 시스템이 통제하는 게 일관적.

# 이슈 #6: 서버시작.bat 중복 실행 시 로딩 지연 (2026-04-24)

## 증상
- `서버시작.bat` 을 두 번 이상 실행하면 콘솔이 여러 개 열림.
- 브라우저에서 `localhost:3000` 로딩이 오래 걸림.
- 한 번 성공하면 그 뒤로는 빠름 — 하지만 bat 을 또 실행하면 재발.

## 원인
- 첫 bat 실행 → Node 프로세스가 포트 3000 / 3001 점유.
- 두 번째 bat 실행 → 새 Node 프로세스가 `EADDRINUSE` 로 바인딩 실패하거나, 대기/재시도하며 응답 불가 상태.
- 두 프로세스가 공존하면 브라우저 요청이 어느 쪽으로 갈지 불확실 → 타임아웃 체감.

## 해결 (2026-04-24)
`서버시작.bat` 시작 시 **포트 3000 / 3001 점유자 강제 종료** 루틴 추가:
```bat
for %%P in (3000 3001) do (
  for /f "tokens=5" %%A in ('netstat -ano ^| findstr ":%%P " ^| findstr LISTENING') do (
    taskkill /F /PID %%A >nul 2>&1
  )
)
```
포트 해제 후 0.5초 대기 → Node 시작 → 브라우저 자동 오픈.
기존 서버가 있어도 자동 교체되므로 **아무 때나 bat 을 눌러도 안전**.

## 재발 방지 체크리스트
- 서버 로딩이 비정상으로 오래 걸리면 → 작업관리자에서 `node.exe` 여러 개인지 확인
- 수동 kill: `taskkill /F /IM node.exe` (모든 노드 종료)
- 포트 확인: `netstat -ano | findstr :3000`

## 교훈
- **포트 바인딩은 독점 자원**. 배치 스크립트가 포트 기반 서버를 구동한다면 **시작 전 기존 점유자 해제** 루틴이 필수.
- "한 번 성공하면 빠르다"는 증상은 대부분 **초기화 불충분** (캐시·포트·잠금 파일 등).
